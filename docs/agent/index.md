# Agent 工程实践

## 1. 什么是 Agent

Agent 是基于大语言模型的任务执行系统。

相比普通 LLM 调用：

    用户输入

    ↓

    Prompt

    ↓

    LLM

    ↓

    回答

Agent 增加了：

-   任务规划；
-   状态管理；
-   工具调用；
-   多步骤执行；
-   结果反馈。

核心思想：

    LLM
    +
    Workflow
    +
    Tools
    +
    Memory

让模型具备完成复杂任务的能力。

------------------------------------------------------------------------

# 2. Agent 基础架构

一个完整 Agent 系统通常包含：

## 用户交互层

负责：

-   请求接收；
-   参数校验；
-   会话管理。

## 编排层

负责：

-   任务拆解；
-   Agent路由；
-   工作流控制。

## 推理层

负责：

-   意图理解；
-   计划生成；
-   结果总结。

## 工具层

负责连接外部能力：

-   数据库查询；
-   搜索；
-   API调用；
-   企业系统。

## 记忆层

保存：

-   当前上下文；
-   历史任务；
-   用户信息。

------------------------------------------------------------------------

# 3. Agent执行模式

## Workflow Agent

通过固定流程控制任务。

例如：

    用户问题

    ↓

    意图识别

    ↓

    检索

    ↓

    分析

    ↓

    生成答案

特点：

优点：

-   稳定；
-   可控；
-   容易上线。

适合：

-   企业应用；
-   工业场景；
-   客服系统。

------------------------------------------------------------------------

## Plan-Execute Agent

流程：

    用户任务

    ↓

    LLM生成计划

    ↓

    执行步骤

    ↓

    汇总结果

特点：

比固定Workflow更加灵活。

适合：

-   复杂任务；
-   多步骤分析。

------------------------------------------------------------------------

## ReAct Agent

核心：

Reasoning + Acting

流程：

    思考

    ↓

    调用工具

    ↓

    观察结果

    ↓

    继续推理

优势：

-   自主性强；
-   适应复杂任务。

问题：

-   成本较高；
-   行为不可预测；
-   需要更多约束。

------------------------------------------------------------------------

# 4. LangGraph设计

LangGraph通过图结构管理Agent流程。

核心概念：

## State

保存任务状态：

``` python
{
    "query": "",
    "context": [],
    "tool_result": {},
    "answer": ""
}
```

## Node

执行具体任务：

例如：

-   检索节点；
-   分析节点；
-   工具节点。

## Edge

控制流程：

包括：

-   固定边；
-   条件边。

------------------------------------------------------------------------

# 5. Agent Memory设计

## 短期记忆

保存当前任务上下文。

例如：

-   当前问题；
-   中间结果；
-   工具返回。

实现：

-   LangGraph Checkpoint。

------------------------------------------------------------------------

## 长期记忆

保存跨会话信息：

例如：

-   用户偏好；
-   历史记录；
-   业务数据。

------------------------------------------------------------------------

## Context Compression

长上下文场景需要压缩。

常见策略：

### 滑动窗口

保留最近消息。

### 摘要压缩

提取：

-   当前任务；
-   关键事实；
-   未完成事项。

### 任务定制压缩

不同任务保留不同信息：

例如：

代码任务：

-   项目结构；
-   文件路径；
-   修改位置。

知识问答：

-   核心事实；
-   用户需求。

------------------------------------------------------------------------

# 6. Tool Calling 与 MCP

Agent通过工具扩展能力。

流程：

    用户请求

    ↓

    LLM判断能力

    ↓

    选择Tool

    ↓

    执行

    ↓

    返回结果

    ↓

    继续处理

工具类型：

-   数据库；
-   搜索服务；
-   企业API；
-   文件处理。

MCP提供统一工具协议，使模型能够标准化调用外部能力。

# 7. Agent工程化设计

## 失败重试

针对：

-   模型异常；
-   网络错误；
-   工具失败。

策略：

-   重试机制；
-   超时控制；
-   降级模型。

------------------------------------------------------------------------

## 输出约束

避免模型输出不可控：

使用：

-   Pydantic；
-   JSON Schema；
-   Structured Output。

------------------------------------------------------------------------

## 成本控制

主要方式：

-   简单问题规则拦截；
-   模型分级调用；
-   Prompt优化；
-   Context压缩。

------------------------------------------------------------------------

## Agent评测

关注：

### 任务完成率

是否完成目标。

### 工具调用成功率

是否正确选择工具。

### 输出质量

包括：

-   准确性；
-   完整性；
-   一致性。

------------------------------------------------------------------------

# 8. 工业和企业场景中的Agent设计原则

实际业务中，并非越自治越好。

企业Agent通常采用：

    确定性Workflow

    +

    LLM语义能力

    +

    工具调用

    +

    人工审核

原因：

-   业务流程明确；
-   需要可解释；
-   需要稳定运行。

------------------------------------------------------------------------

# 9. 总结

Agent工程的核心不是简单调用大模型，而是围绕业务目标设计：

-   合理的任务流程；
-   稳定的状态管理；
-   可控的工具调用；
-   有效的记忆机制；
-   完善的评测和治理体系。

从 Demo 到生产级 Agent，关键在工程化能力。
