# 大模型基础与工程实践

## 1. 大模型应用需要掌握什么

大模型应用工程不等于只会调用 API。

需要理解：

- Transformer 基础结构；
- Token、Embedding 与上下文表示；
- Attention 与自回归生成；
- Prompt 与结构化输出；
- 微调、LoRA 和蒸馏；
- 推理成本与稳定性；
- 模型能力边界。

---

## 2. Transformer 基础

Transformer 通过自注意力机制建模序列中不同位置之间的关系。

核心结构：

```text
输入 Token
  ↓
Embedding
  ↓
位置编码
  ↓
多头注意力
  ↓
前馈网络
  ↓
残差连接与归一化
  ↓
多层堆叠
```

### Encoder 与 Decoder

Encoder 更适合理解完整上下文，BERT 是典型 Encoder-only 模型。

Decoder 使用因果掩码进行自回归生成，GPT 类模型属于 Decoder-only。

---

## 3. Attention

缩放点积注意力：

```text
Attention(Q, K, V)
=
Softmax(QKᵀ / √dₖ)V
```

含义：

- Q：当前 Token 想查找什么；
- K：其他 Token 提供什么匹配特征；
- V：匹配后实际聚合的信息。

除以 `√dₖ` 是为了控制点积数值规模，避免 Softmax 过早进入饱和区。

### 多头注意力

不同注意力头可以关注：

- 语法关系；
- 指代关系；
- 长距离依赖；
- 局部模式。

各头结果拼接后，再通过输出投影矩阵融合。

---

## 4. 因果掩码

GPT 生成时不能看到未来 Token。

实现方式：

```text
QKᵀ 得分矩阵
+
上三角 -∞ 掩码
+
Softmax
```

未来位置对应概率被压到接近 0，从而实现从左到右生成。

---

## 5. 位置编码与 RoPE

模型本身不能感知 Token 顺序，因此需要位置编码。

常见演进：

```text
绝对位置编码
  ↓
相对位置编码
  ↓
RoPE 旋转位置编码
```

RoPE 主要作用在 Q 和 K 上，使注意力分数自然包含相对位置信息。

---

## 6. BERT 与 GPT

### BERT

特点：

- 双向上下文；
- Encoder-only；
- 适合理解类任务。

常见应用：

- 文本分类；
- 实体识别；
- 语义相似度；
- Embedding；
- RAG 召回。

### GPT

特点：

- 因果掩码；
- Decoder-only；
- 自回归生成。

常见应用：

- 对话；
- 内容生成；
- 代码生成；
- Agent 推理；
- 工具调用。

---

## 7. Embedding

Embedding 把 Token、句子或文档映射到向量空间。

相似语义在向量空间中距离更近。

在前向传播中：

- 原始词表 Embedding 是模型参数；
- 每一层隐藏状态会融合上下文；
- 同一个词在不同语境下得到不同上下文表示。

RAG 使用的文本向量通常来自专门训练的 Embedding 模型，而不是直接取生成模型最后一层输出。

---

## 8. MoE

混合专家模型由：

- 多个专家网络；
- 门控网络；
- Top-K 路由；
- 负载均衡损失；

组成。

每个 Token 只激活部分专家，因此可以扩大模型总参数量，同时控制单次计算成本。

MoE 的工程问题包括：

- 专家负载不均；
- 路由崩塌；
- 跨设备通信成本；
- 专家并行复杂度。

---

## 9. Prompt Engineering

Prompt 主要用于：

- 定义角色和任务；
- 注入业务规则；
- 提供示例；
- 约束输出格式；
- 指定工具调用规则；
- 明确拒答边界。

生产环境应避免把全部可靠性寄托在 Prompt 上。

Prompt 属于软约束，还需要：

- Schema 校验；
- 参数校验；
- 工具白名单；
- 权限控制；
- 内容安全；
- 超时和重试。

---

## 10. 结构化输出

对于 Agent、信息抽取和后端接口，应优先使用结构化输出。

常见方式：

- JSON Schema；
- Pydantic；
- Function Calling；
- constrained decoding。

结构化输出流程：

```text
LLM 输出
  ↓
Schema 校验
  ↓
失败重试或修复
  ↓
业务规则校验
  ↓
进入后续节点
```

格式正确不代表业务正确，因此还要做语义和规则校验。

---

## 11. LoRA 与 QLoRA

LoRA 冻结原模型权重，在目标层增加低秩矩阵：

```text
W = W₀ + BA
```

只训练 A、B，可以显著减少可训练参数量。

适合：

- 领域适配；
- 输出风格约束；
- 特定任务微调；
- 资源有限场景。

QLoRA 在 LoRA 基础上对基座模型量化，进一步降低显存占用。

LoRA 不适合解决所有知识更新问题。频繁变化的事实知识通常更适合 RAG。

---

## 12. SFT 与后训练

SFT 使用高质量指令数据训练模型学习：

- 任务格式；
- 领域表达；
- 输出结构；
- 推理模式。

数据质量通常比单纯扩大数据量更重要。

需要关注：

- 标签一致性；
- 难例覆盖；
- 数据泄露；
- 训练集和评测集隔离；
- 过拟合；
- 灾难性遗忘。

---

## 13. 知识蒸馏

知识蒸馏使用教师模型帮助学生模型学习。

典型损失：

```text
总损失
=
硬标签损失
+
软标签蒸馏损失
+
业务任务损失
```

软标签包含类别之间的相对关系，比独热标签提供更多信息。

工程上必须确保：

- 教师和学生任务定义一致；
- 输出空间可对齐；
- 软标签经过规则和人工抽检；
- 蒸馏指标与业务目标一致。

大模型推理过程不能简单视为可以直接复制到任意小模型，需要根据任务形式设计可学习的监督信号。

---

## 14. 推理阶段

推理阶段通常关闭：

- 梯度计算；
- 反向传播；
- 优化器；
- Dropout 等训练随机性。

模型生成过程：

```text
输入上下文
  ↓
前向传播
  ↓
得到下一个 Token 的 Logits
  ↓
采样或选择 Token
  ↓
拼接到上下文
  ↓
继续生成
```

常见采样参数：

- temperature；
- top-k；
- top-p；
- repetition penalty；
- max tokens。

结构化任务通常使用较低温度，提高稳定性。

---

## 15. 模型选择

模型选择不应只看榜单。

需要综合：

- 任务准确率；
- 结构化输出能力；
- 上下文长度；
- 首 Token 延迟；
- 吞吐量；
- 调用成本；
- 私有化部署要求；
- 工具调用稳定性；
- 中文领域表现。

可以采用模型分级：

```text
简单分类与路由
  → 小模型

常规问答与抽取
  → 中等模型

复杂推理与高风险任务
  → 强模型

主模型失败
  → 备用模型
```

---

## 16. 大模型应用护栏

软护栏：

- System Prompt；
- 示例；
- 输出要求；
- 拒答规则。

硬护栏：

- 输入校验；
- 输出 Schema；
- 工具白名单；
- 参数消毒；
- 权限校验；
- 最大循环次数；
- Token 和成本限制；
- 敏感操作人工确认；
- 状态持久化和审计日志。

生产级系统需要软硬护栏共同工作。

---

## 17. 总结

大模型应用工程的核心不是追求模型“什么都能做”，而是：

- 正确认识模型能力；
- 选择适合的模型；
- 构建可靠上下文；
- 约束输入输出；
- 设计评测与兜底；
- 把模型嵌入真实业务系统。

模型决定能力上限，工程系统决定能力能否稳定落地。
